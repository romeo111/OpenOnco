# OpenOnco Domain

OpenOnco is a clinical decision-support system that turns a patient profile into one or more treatment plans, each citing the rule-engine reasoning and the primary sources behind it. The domain language below is what every architecture suggestion, code review, and refactor proposal must use. Definitions follow `specs/KNOWLEDGE_SCHEMA_SPECIFICATION.md` (UA, authoritative); the English names match the YAML schema field names.

## Language

### Knowledge-base entities (declarative, version-controlled YAML)

**Disease**:
A clinical condition the system supports (e.g. DLBCL NOS, AML, HCV-MZL). Each Disease aggregates Indications, Algorithms, RedFlags, Workups for that condition.
_Avoid_: diagnosis, illness, condition.

**Biomarker**:
A measurable molecular or cellular feature used in clinical decisions (e.g. TP53 mutation, MYC translocation, PD-L1 ≥ 1%). Biomarkers gate Indications and trigger RedFlags.
_Avoid_: marker, mutation, gene (these are too narrow — Biomarker covers IHC, FISH, sequencing, scoring scales).

**Drug**:
A single pharmacological agent (e.g. rituximab, venetoclax). Drugs are the leaves of Regimens.
_Avoid_: medication, agent, compound.

**Regimen**:
A protocoled combination of Drugs + dosing + schedule + duration (e.g. R-CHOP × 6, BR fixed-duration, ATRA+ATO chemo-free). One Regimen can be reused across Indications.
_Avoid_: protocol, treatment, therapy combo.

**Indication**:
A clinical scenario in which a specific Regimen is the recommended choice — the link between Disease+criteria and Regimen, with evidence and Source citations. The basic unit the rule engine selects from.
_Avoid_: recommendation, rule, prescription.

**Contraindication**:
A condition under which a Drug or Regimen is not safe to use (hard or relative). Filters candidate Indications.
_Avoid_: warning, allergy, exclusion.

**RedFlag**:
A patient-level signal that escalates urgency, changes pathway, or rules out an Indication (e.g. RF-DLBCL-HIGH-IPI, RF-APL-DIC, RF-MM-HIGH-RISK-CYTOGENETICS). Each RedFlag has a name, fire condition, and the rule-engine action it triggers.
_Avoid_: alert, warning, flag.

**Test**:
A diagnostic procedure the patient may need or has had (e.g. flow cytometry, FISH for t(11;14), bone-marrow biopsy). Used by Workups and Algorithms.
_Avoid_: assay, exam, investigation.

**SupportiveCare**:
A non-anti-cancer intervention that accompanies a Regimen (e.g. TLS prophylaxis, HBV prophylaxis, growth factor support). Attached to Indications.
_Avoid_: ancillary care, support.

**MonitoringSchedule**:
A timetable of follow-up Tests and assessments after a Regimen starts.
_Avoid_: follow-up, surveillance.

**Algorithm**:
A declarative branching pathway that maps Disease + patient state + RedFlags to one or more Indications. The rule-engine traversal target. Has versioned steps and outcomes.
_Avoid_: flowchart, decision tree, pathway.

**Workup**:
A set of Tests recommended pre-treatment to characterise the Disease (e.g. lymphoma diagnostic workup before histology). Distinct from MonitoringSchedule (post-treatment).
_Avoid_: triage, evaluation.

**Source**:
A citable external document — published guideline (NCCN, ESMO, ELN), trial publication, regulatory label (FDA, EMA), open database (ClinicalTrials.gov, DailyMed). Every clinical claim in an Indication or RedFlag must cite ≥ 1 Source. Sources are mostly `referenced` (linked, not hosted); `hosted` requires explicit H1–H5 license justification.
_Avoid_: reference, citation, doc.

### Runtime concepts (engine output, not stored as YAML)

**Patient**:
The structured input — demographics, Disease at code-systems level (ICD-10/O-3 morphology), Biomarkers measured, comorbidities, prior treatments, performance status. FHIR R4/R5 + mCODE on the wire.
_Avoid_: case, profile, record (those are presentation layers).

**Plan**:
The engine's output for a given Patient — typically two parallel tracks (default + alternative) with the chosen Indication, Regimen, SupportiveCare, fired RedFlags, and citations. Generated by `generate_plan`.
_Avoid_: recommendation, output, result.

**DiagnosticBrief**:
The engine's output for a pre-biopsy Patient — a Workup brief, not a Plan, since CHARTER §15.2 C7 forbids treatment selection without histology. Generated by `generate_diagnostic_brief`.
_Avoid_: pre-plan, draft plan.

**MDT** (Multi-Disciplinary Team output):
A skill-based commentary layer attached to a Plan or DiagnosticBrief — pathologist note, radiologist note, etc. Not the clinical decision; it surrounds it. Generated by `orchestrate_mdt`.
_Avoid_: tumor board, panel, review.

## Relationships

- A **Disease** has many **Indications**, **Algorithms**, **RedFlags**, **Workups**, **MonitoringSchedules**.
- An **Indication** binds exactly one **Regimen** to a (Disease + criteria) and cites ≥ 1 **Source**; can have many **SupportiveCare** items and **Contraindications**.
- A **Regimen** is composed of one or more **Drugs**.
- An **Algorithm** traverses **RedFlags** and patient state to select **Indications**.
- A **RedFlag** fires from one or more **Biomarkers** and/or patient-state predicates and cites ≥ 2 **Sources**.
- A **Workup** lists **Tests** the patient should complete pre-treatment.
- A **Patient** + KB → engine produces a **Plan** (or a **DiagnosticBrief** if no histology) + **MDT** commentary.

## Example dialogue

> **Dev:** "When a **Patient** has a confirmed **Disease**, the engine runs the matching **Algorithm**, fires any **RedFlags** from the patient's **Biomarkers**, and returns a **Plan** with a default and alternative **Indication**. Each **Indication** carries its **Regimen** and **Source** citations."
> **Clinical Co-Lead:** "Yes — and if the Patient has no biopsy yet, we don't return a **Plan**; we return a **DiagnosticBrief** that lists the **Workup**'s **Tests**. The engine will not choose a **Regimen** before histology — that's CHARTER §8.3."

## Flagged ambiguities

- "Algorithm" is overloaded: in CS it means any procedure, in OpenOnco it specifically names a versioned **Algorithm** entity (a clinical decision pathway). Inside engine code, prefer `pathway`, `traversal`, or `step` for non-entity uses.
- "Source" vs "reference" vs "citation" — pick **Source** for the entity, "citation" for the act of pointing to it from an Indication or RedFlag, and avoid "reference" entirely (overlaps with the `referenced` license-classification value).
- "Plan" is also used loosely to mean "the output document" in user-facing copy. Inside the codebase, **Plan** is the structured engine output; the rendered HTML is the `plan-render` artifact, not a Plan.
- "Workup" (pre-treatment Tests) vs "MonitoringSchedule" (post-treatment Tests) — both are sets of Tests; the temporal phase is what distinguishes them. Don't merge.
