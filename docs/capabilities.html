Capabilities

OpenOnco is a declarative rule engine for oncology recommendations. Input: a JSON patient profile (FHIR R4 / mCODE-compatible). Output: a Plan with ≥2 alternative tracks (standard + aggressive), or a DiagnosticPlan with a workup brief if histology isn't yet confirmed. Every claim ships with a citation; every algorithm step is traced. This page is a complete, honest description of what we do, and at the bottom — what we deliberately don't do, where the clinician remains the final authority.

STUB status across all clinical content. Reviewer sign-offs ≥ 2: 15/317 (CHARTER §6.1 requires two Clinical Co-Lead approvals for any Indication before it can be considered «published»). Right now this is a proposed plan: structured data + algorithm + sources are in place, but without dual sign-off. State as of 2026-04-30 20:12 UTC. This is a decision-support tool, not a medical device.

0. What's new this past week

The KB and engine picked up several structural updates over the past few days — they change how the engine works, not only what it knows:

CIViC
Actionability — full OncoKB replacement

Migrated from closed OncoKB (incompatible with CHARTER §2 — non-commercial) to CIViC (CC0). The engine reads a nightly YAML snapshot via SnapshotCIViCClient; a fusion-aware matcher handles both point mutations and fusions (BCR::ABL1 etc.). Render is ESCAT-primary with CIViC-detailed deep-dive. Monthly snapshot refresh in CI + diff-only update so nothing drifts silently.

Phases
Multi-phase regimens (PR1-3)

Regimen.phases decomposes a course into ordered, named blocks (induction → consolidation → maintenance). bridging_options lists bridging-regimen IDs (e.g. for CAR-T). Render is phases-aware: every phase renders with its own cycle schedule. Back-compat: legacy single-phase YAMLs auto-wrap to one-phase form.

Guard
Citation-presence guard at HTML render

No BMA claim ships in HTML without an explicit citation. A render-time guard checks every cell in the actionability table; missing-source rows get a warn-badge or are dropped in strict mode. This is Layer 3 of a three-tier system — preceded by a background citation verifier (3-layer grounding for all sidecar PRs) and loader-level SRC-* referential integrity.

Matrix

Per-disease table: bio / drug / ind / reg / rf counts, 1L+2L checkmarks, questionnaire status, fill% + verified%. Grouped by lymphoid heme, myeloid heme, solid tumours; family-level avg metrics. The canonical UI surface for disease_coverage.json. → See the matrix

434

Structured clinical conditions that automatically rewrite the plan: RF-BULKY-DISEASE (nodal mass >7 cm) flips HCV-MZL from antiviral-first to BR + DAA; RF-MM-HIGH-RISK-CYTOGENETICS (t(4;14), del(17p), gain 1q) escalates MM from triplet VRd to quadruplet D-VRd. 2. How a request is processed — 6 engine stages

The clinician hands the engine a JSON profile (FHIR/mCODE-compatible in the future, simplified dict in MVP). The engine runs 6 sequential stages and returns a Plan with ≥2 alternative tracks (CHARTER §2 — both tracks in one document, alternative not hidden).

Stage 1
Disease + Algorithm resolve
disease.icd_o_3_morphology or disease.id → find Disease entity. Disease + line_of_therapy + disease_state → find Algorithm.
Stage 2
Findings flattening
Merges demographics + biomarkers + findings into one flat dict for evaluation.
Stage 3
RedFlag evaluation
Each of 434 RFs is checked against findings. Boolean engine: any_of/all_of/none_of clauses with thresholds. The RF-trigger alias map cut «unevaluated» warnings by ~76%.
Stage 4
Algorithm walk
Decision tree step by step. Each step → outcome → branch (result or next_step). Trace stores all fired_red_flags at each step.
Stage 5
Tracks materialization
All Indications from algorithm.output_indications become separate tracks (standard / aggressive / surveillance). A biomarker-aware track filter drops tracks that violate biomarker_requirements_excluded.
Stage 6
Per-track resolution
Indication → Regimen (with phases) → MonitoringSchedule + SupportiveCare + Contraindications + AccessMatrix row. CIViC actionability hits — separate detail section.

Per-patient processing time is 50–200 ms (KB load dominates). In Pyodide the first run is 8–15 s (runtime download); subsequent runs are like a local CLI. There is no server — the engine runs locally (CLI) or in the user's browser (Pyodide). The patient JSON never leaves the machine.

3. Coverage matrix — public progress picture

The page /diseases.html is a per-disease table that shows what's in the KB for each of 65 diseases. Grouped into three families: lymphoid heme, myeloid heme, solid tumours. Why a matrix: publicly shows how «alive» each disease is — so the clinician doesn't rely on «X must be in there somewhere» and immediately sees whether we have a 2L algorithm for this case or not. For contributors — the matrix shows where the biggest gaps are and which work to take first.

4. CIViC actionability — how we decide «what to do with the biomarker»

Until recently, actionability (the evidence level for biomarker→drug relationships) came from OncoKB. The OncoKB ToS forbids non-commercial public derivatives — a conflict with CHARTER §2 (free public resource). We migrated to CIViC (CC0) — Clinical Interpretation of Variants in Cancer, the open WashU resource. How it works now:

snap
Nightly YAML snapshot

The engine doesn't call the CIViC API at runtime — it reads a local snapshot from knowledge_base/hosted/civic/<date>/. This guarantees deterministic results and works offline. SnapshotCIViCClient is the public interface.

match
Fusion-aware variant matcher

civic_variant_matcher handles both point mutations (BRAF V600E) and fusions (BCR::ABL1 — both components indexed) and structural variants. Fusion-agnostic queries («ABL1») and component-specific queries map identically.

render
ESCAT-primary, CIViC-detailed

In Plan render: ESCAT tier (I-A, II-B, …) — primary signal, eye-level. CIViC evidence rating (A/B/C/D, supports/does-not-support) + clinical-significance — in a details accordion. No OncoKB-level fields in render.

CI
Monthly snapshot refresh + diff CI

civic-monthly-refresh.yml in GitHub Actions pulls a new CIViC dump monthly, regenerates the snapshot, computes a diff against the previous version. If a new snapshot changes N entities — opens a PR with a diff checklist for review. No surprise recommendation drift from silent upstream changes.

5. Multi-phase regimens — induction → consolidation → maintenance

Before the PR1-3 phases-refactor, a regimen was a flat list of drugs with one cycle schedule. Now Regimen.phases is an ordered list of named blocks, each with its own component list, cycles, duration, and trigger for transitioning to the next phase. This properly models real protocols: R-CHOP × 6 → maintenance, AML 7+3 → HiDAC consolidation, axi-cel bridging → infusion → preemptive tocilizumab maintenance.

Regimen.bridging_options is a list of bridging-regimen IDs between phases (e.g. for CAR-T: bridging chemo between apheresis and infusion). Render is phases-aware: each phase renders as a separate block with its own cycle schedule, premedications, monitoring.

Back-compat: legacy single-phase YAMLs (with components: at top level and no phases:) auto-wrap to one-phase form via auto_wrap_legacy_components() at load. One-way only: if the author wrote phases: explicitly, we don't touch components. 18 high-stakes regimens migrated manually (PR3).

6. Citation guard — nothing ships in HTML without a source

A three-layer citation-verification system — three points where we catch sourceless claims:

LayerWhere it catchesWhat it does
Layer 1 — Loader YAML load time (Pydantic) Checks SRC-* referential integrity: every source_id in Indication/Regimen/RedFlag must resolve to a Source entity. Unknown SRC-ID → load fail.
Layer 2 — Background verifier Sidecar PR audit (CI) citation-verifier: three-layer grounding for each AI-generated claim in a sidecar — does the source exist, is the paraphrase grounded in the original text, do the claim-anchor coordinates fall inside the cited section. Runs on every contributor PR.
Layer 3 — Render guard HTML output time _citation_guard: at render time, checks every BMA cell. Missing citation → warn-badge in lenient mode or cell skip in strict_citation_guard=True. Guarantee: what you see in HTML has a source.

7. How we read the patient profile

The engine only reads structured fields from the patient profile. Every field has clear semantics: it either fires a RedFlag, filters available Indications, or configures Regimen materialization. Unknown fields are ignored — no «hidden effects».

CategoryWhat we readHow we use it
Disease (entry point) disease.id · icd_o_3_morphology · line_of_therapy · disease_state determines which Algorithm to run
Diagnostic-mode trigger disease.suspicion.lineage_hint · tissue_locations · presentation switches to DiagnosticPlan instead of Plan (workup brief)
Demographics age · sex · ecog · fit_for_transplant · decompensated_cirrhosis · pregnancy_status filter in Indication.applicable_to.demographic_constraints
Biomarkers any BIO-X from KB as keys: BIO-CLL-HIGH-RISK-GENETICS, BIO-MM-CYTOGENETICS-HR, BIO-HCV-RNA, ... fire RedFlags, filter Indications, map to CIViC actionability
Findings hundreds of structured fields — dominant_nodal_mass_cm, ldh_ratio_to_uln, creatinine_clearance_ml_min, blastoid_morphology, tp53_mutation, del_17p, ... thresholds in RedFlag triggers
Prior tests completed prior_tests_completed: [TEST-IDs] excludes already-done tests from generated workup_steps
Clinical record (free-form) any clinical_record envelope not read by the engine — used only by render layer for context

8. Three ways to run the engine — none server-side

CLI
Locally on the clinician's machine

python -m knowledge_base.engine.cli --patient profile.json --render plan.html. Works offline, no network needed. The profile stays on disk.

Pyodide
In the browser (try.html)

Pyodide v0.26.4 loads a Python WebAssembly runtime, micropip installs pydantic+pyyaml, the engine bundle (~2.4 MB) is unpacked into in-memory FS. The engine runs in the browser. The patient JSON never leaves the machine. A service worker caches the bundle for offline use.

Library
Python import

from knowledge_base.engine import generate_plan, revise_plan — integration with EHR, CSV pipelines, batch testing. Stateless, deterministic.

Privacy by design. The patient JSON never leaves the user's machine. No logs, no DB, no accidental leakage. Reproducibility: Plan.knowledge_base_state.algorithm_version pins the KB version → same input + same KB = same output.

10. How the plan updates as new data arrives

revise_plan(updated_patient, previous_plan, revision_trigger) takes an updated profile and produces a new plan version with a supersedes/superseded_by chain. Three legal transitions plus one prohibition:

FromWith changeTransitionResult
DiagnosticPlan vNonly suspicion (no histology)diagnostic → diagnosticDiagnosticPlan v(N+1)
DiagnosticPlan vNconfirmed histologydiagnostic → treatment (promotion)Plan v1 (first treatment)
Plan vNany update with histologytreatment → treatmentPlan v(N+1)
Plan vNhistology removedILLEGAL — ValueError, CHARTER §15.2 C7

The previous plan is not mutated — a deep copy is returned with superseded_by set. The caller (CLI / EHR) decides what to do with both versions. Per CHARTER §10.2 — the old version is retained indefinitely.

Boundaries — what the engine deliberately doesn't do

Knowing the boundaries matters as much as knowing the capabilities. The sections below are a complete and honest list of where the engine refuses to generate decisions without data, where the clinical call remains with the clinician, and which architectural positions we do not plan to change.

12. Open Questions — how the engine refuses to decide without data

The engine does not make decisions without the data it needs. Instead of silently defaulting, it explicitly records which fields are missing and which test or finding is required. This is the Open Questions mechanism — part of MDT-orchestrator (Q1-Q6 + DQ1-DQ4 rules per MDT_ORCHESTRATOR_SPEC §3).

Treatment-mode Open Questions (Q1-Q6) — examples from real code

  • Q1 — Histology not confirmed: if disease.id resolves but there's no biopsy_date or histology_report — emit warning «Treatment Plan generated against ICD-O-3 code only; recommend confirming primary histology before initiating therapy».
  • Q2 — Stage missing: if Algorithm.decision_tree references staging but the profile has no stage — fall through to default with flag «Lugano/Ann Arbor stage required for confident risk-stratification».
  • Q3 — RedFlag clause references absent findings: if RF-MM-HIGH-RISK-CYTOGENETICS checks tp53_mutation + del_17p + t_4_14 + gain_1q, and the profile only has del_17p — engine doesn't false-negative; emit «Cytogenetic panel incomplete; high-risk status assessed with partial data».
  • Q4 — Biomarker required by Indication missing: if IND-CLL-1L-VENO requires BIO-CLL-HIGH-RISK-GENETICS for default-track selection — emit «IGHV mutation status + FISH del(17p) required to confirm 1L recommendation».
  • Q5 — Performance status missing: if ecog is absent — fall to a conservative default (only standard track), emit «ECOG performance status required for transplant-eligibility assessment».
  • Q6 — Drug availability flag: if the selected Regimen contains a drug marked nszu_reimbursement: false (e.g. daratumumab in MM) — emit «D-VRd: daratumumab not currently NSZU-reimbursed in Ukraine; verify funding pathway before initiation».

Diagnostic-mode Open Questions (DQ1-DQ4) — for pre-biopsy mode

  • DQ1 — Tissue location missing: if suspicion.tissue_locations is empty — workup match can't rank, emit «Tissue location must be provided for workup matching».
  • DQ2 — Lineage hint absent: without lineage_hint the engine uses only tissue + presentation for matching, lower confidence.
  • DQ3 — Presentation free-text empty: presentation_keywords scoring × 0; only lineage + tissue contribute.
  • DQ4 — Working hypotheses not provided: the engine has no preferred direction; the most generic workup wins.
Why not «default silently»: CHARTER §15.2 C6 (anti automation-bias) — the engine cannot pretend to know when it doesn't. Every missing-data situation must be visible to the clinician. Open Questions render in the Plan as a separate section, not hidden.

13. What the engine deliberately doesn't do — personalization gaps

«Personalization» in OpenOnco is rule-based selection from fixed options, not AI generation. This is a deliberate architectural position (CHARTER §8.3). Concrete gaps:

Gap 1

No per-patient dose calculation

The Regimen stores a standard dose (bortezomib 1.3 mg/m²); we don't multiply by the patient's BSA or auto-reduce for CrCl 30 mL/min. The clinician recalculates. This is intentional, to avoid FDA-medical-device classification.

Gap 2

No response-adapted cycle adjustment

Regimen pins total_cycles: 6 + 2 maintenance. The engine doesn't auto-adapt based on response (PR vs CR after PET2). A re-staging plan is generated via a separate revise_plan with a new profile — the clinician triggers it explicitly.

Gap 3

Genomic matching limited to curated biomarkers

CIViC covers most actionable variants. Outside CIViC (rare, novel, unverified variants) the engine emits a warning «no actionability evidence in current snapshot» — never «creative» interpretation. This is a coverage limit, not engine logic.

Gap 4

SupportiveCare uniform per regimen

PJP prophylaxis is attached to D-VRd for everyone — even a patient with bactrim allergy. The engine doesn't know alternatives (dapsone instead of bactrim). The clinician substitutes manually.

Gap 5

No cumulative-toxicity tracking across lines

2L+ algorithms exist for 34 heme diseases (133 indications 2L+), but the profile doesn't carry prior_treatment_history as a structured field. A 2L plan for a patient who got bortezomib in 1L with grade 2 neuropathy — the engine doesn't know about prior exposure unless explicitly stated; the clinician interprets prior_lines from free text.

14. CHARTER constraints — will not change

These are not technical debt — they are principled architectural decisions that position the project as non-device CDS and gate FDA / clinical safety.

CHARTER §8.3

The LLM does not make clinical decisions

LLMs help only with: boilerplate code, doc drafts, extraction from clinical documents (with human verification), translation with clinical review. Not: regimen selection, dose generation, biomarker interpretation for therapy choice.

CHARTER §15.2 C7

No histology — no treatment Plan

A treatment Plan is generated only if disease.id or icd_o_3_morphology is confirmed. Otherwise the engine refuses and switches to DiagnosticPlan mode. revise_plan from treatment back to diagnostic is forbidden and raises ValueError.

CHARTER §15.2 C5

No time-critical recommendations

The engine isn't designed for emergency oncology (oncologic emergencies, time-sensitive infusion reactions). That would trigger device classification. If an Indication is marked time_critical: true — the engine adds a disqualification warning to FDA compliance.

CHARTER §6.1

Two-reviewer merge for clinical content

Any change under knowledge_base/hosted/content/ that affects clinical recommendations needs two of three Clinical Co-Lead approvals. Without that the Indication stays STUB.

CHARTER §15.2 C6

Anti automation-bias mandatory

The engine never shows only one recommendation — always ≥2 tracks side-by-side. Alternative is not buried, not «click to expand», not fine-print. The clinician sees this as a choice, not a directive.

CHARTER §9.3

Patient data never in repo / public artifact

patient_plans/ is gitignored. Any patient HTML — gitignored pattern. The site (docs/) shows only synthetic examples. Telemetry collection is forbidden without explicit consent.

16. What the engine never does — explicit boundary list

Never

Never hides alternative track

Both recommendations always shown. UI has no «expand to see alternative» pattern.

Never

Never invents an Indication via LLM

Everything is selected from the curated KB. No matching Indication → emit warning, no «creative invention».

Never

Never modifies doses «for the patient»

Doses are standard NCCN/ESMO. Adjustments only via explicit dose_modification_rules in Regimen YAML, no ad-hoc calculations.

Never

Never judges «which is better» between tracks

The Algorithm picks default but doesn't decide that default is «better». The clinician has full autonomy to pick alternative — documented in automation_bias_warning.

Never

Never interprets imaging

«Bulky disease» comes as a structured field dominant_nodal_mass_cm, not from image analysis. Image analysis = device classification.

Never

Never does cohort matching

«N patients in our DB chose X» — a separate future feature, requires a persisted patient registry + privacy review. Currently unavailable.

17. How to live with this — clinician workflow

This engine is designed as preparation for a tumour board, not a replacement. The clinician feeds in a profile, gets a structured draft with all sources and open questions, and then:

1
Verifies sources

Every claim in the plan has a citation. The clinician can read the original NCCN/ESMO/MoH section and confirm the engine didn't misquote. The citation guard already filtered out sourceless cells.

2
Closes Open Questions

If the engine emits «cytogenetic panel incomplete» — the clinician orders the test, adds it to the profile, runs revise_plan. The plan updates, the OpenQuestion closes.

3
Adapts to the patient

Re-checks doses, substitutes supportive care for allergies, verifies Ukraine availability manually. The engine is the draft, the clinician is the final.

4
Tumor board discusses

The MDT brief shows which roles activated and which questions are open. It's a structured agenda for a board meeting. Decisions from the board are pinned as provenance events.

18. How to contribute — verify algorithms with your AI tokens

The biggest gap on this page — 15/317 dual-reviewer sign-off. This bottleneck isn't solved by more code; it's solved by contributors with AI tools who take structured chunks of work and execute each in their own worktree. We call this TaskTorrent.

1
What it is

The maintainer publishes a «chunk» — one concrete, complete task (~100k–300k tokens of structured AI work): «re-verify BMA evidence for DLBCL × 17 entities» or «backfill source-license for 8 sources». The contributor takes the chunk, the AI agent does the work, opens a PR.

2
What you need

An AI tool with token budget per chunk (Claude Code Pro+, Codex, Cursor, ChatGPT with web access). GitHub account + gh CLI. Python 3.10+. ~1–3 hours. No clinical expertise — you don't write medical advice; you trigger structured drafting, and the clinical co-leads then sign off.

3
8-line bootstrap

Copy the 8-line prompt from /contribute.html into your AI agent. It will find the next available chunk on its own, read the spec, claim it, do the work under contributions/<chunk-id>/, run the validator, open a PR.

4
What happens to the PR

The maintainer reviews the mechanical part (validator, schema integrity). A Clinical Co-Lead samples the semantic part (CHARTER §6.1). After merge — the sidecar lands in master, then upserts into hosted KB, then renders for patients. Attribution lives in _contribution_meta.yaml (ai_tool + ai_model).
